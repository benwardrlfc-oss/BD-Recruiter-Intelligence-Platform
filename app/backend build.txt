6. User CSV upload architecture

Users must be able to upload CSV company lists to the platform.

This feature should work safely and cleanly.

CSV upload goals

allow users or organisations to import their own target company lists

enrich the platform over time

match uploaded rows to existing canonical companies

create new company candidates where needed

preserve provenance of who uploaded what

improve the central data store without polluting it with junk

CSV upload flow
Step 1 – upload and staging

Upload CSV into a staging table / staging object, not directly into canonical tables.

Store:

uploader user_id

organisation_id

original file

upload timestamp

row count

raw row contents

Step 2 – schema mapping

Support flexible column mapping such as:

company name

website

city

country

sector

ownership

notes

custom tags

Users should be able to map columns to known system fields where needed.

Step 3 – validation

Validate:

required fields present

formats are acceptable

websites / domains are plausible

duplicate rows inside the upload are flagged

row-level errors are logged

Step 4 – entity resolution

For each row:

try to match to existing canonical company

use exact and fuzzy matching

use website / domain matching

use geography / ownership clues

produce match confidence

Possible states:

matched canonical company

possible duplicate / ambiguous

new company candidate

Step 5 – enrichment

Use uploaded data to enrich existing records where appropriate, but do not blindly overwrite trusted fields.

Use source hierarchy and field confidence.

Step 6 – provenance and ownership

Track:

uploaded_by_user

uploaded_by_org

source_type = user_csv

whether the data created a new company, enriched an existing one, or was rejected

Step 7 – review queue where needed

Low-confidence or conflicting rows should go to a review state rather than corrupt the canonical registry.

Important rule

User CSV uploads should be able to improve the system, but never degrade data quality.
